### Q1：在 Scala REPL 中，res 变量是 val 还是 var？

**答：res 变量是 `val`（不可变引用）。**

Scala REPL（交互式解释器）每次执行表达式后，自动用 `res0`、`res1`……保存结果。这些变量都是 `val`，即**不可重新赋值**，永远指向第一次计算的结果。

> 💡 源码佐证：`code3_2.scala` 中 `object HelloWorld` 内的变量都是用 `val`/`var` 显式声明，REPL 的自动变量行为与此一致。

### Q6：什么是闭包？

**答：闭包是一个函数，其返回值依赖于函数外部声明的自由变量。** 这个函数会"捕获"并记住该外部变量的**引用**（而非当时的值）。

示例：

```scala
var factor = 2
val multiplier = (i: Int) => i * factor   // factor 是自由变量，形成闭包
factor = 3
println(multiplier(5))   // 输出 15（捕获的是变量的引用）
```

> 💡 源码示例：`code3_67.scala`、`code3_68.scala` 中演示了 `_ + _` 作为匿名函数简写，本质上也是闭包。

### Q7：Scala 函数中把方法体的最后一行作为返回值，需不需要显式调用 return？

**答：不需要。** Scala 自动将方法体的最后一行表达式的值作为返回值。显式使用 `return` 反而需要显式声明返回类型，且可能改变控制流（如嵌套函数中），因此不推荐。

```scala
def add(x: Int, y: Int): Int = {
  x + y      // 自动作为返回值
}            // 无需 return
```

### Q8：map 和 flatMap 有什么区别？请举例说明。

| 操作 | 映射关系 | 元素个数 | 用途 |
|:---|:-------|:-------|:----|
| **`map`** | 一对一 | 个数不变 | 每个元素单独变换 |
| **`flatMap`** | 一对多 + 拍平 | 可能变化 | 拆分+展平（如 WordCount） |

示例：

```scala
val list = List("Hello", "World")

// map：每个字符串变为字符数组，结构不变
val mapped = list.map(_.toCharArray)
// 结果: List(Array(H,e,l,l,o), Array(W,o,r,l,d))

// flatMap：每个字符数组展平为单个字符集合
val flatMapped = list.flatMap(_.toCharArray)
// 结果: List(H, e, l, l, o, W, o, r, l, d)
```

|:--------|:------------------:|:--------|
| **重写抽象方法** | 必须加（可省略但强烈建议加） | 父类无实现，子类必须提供 |

### Q4：定义抽象类时的注意事项

- 使用 `abstract class` 声明，**不能被实例化**（不能 `new`）
- 可以包含**抽象字段**（未初始化的 `val`/`var`）和**抽象方法**（无方法体）
- 也可以包含**具体实现**（已初始化的字段和已实现的方法）
- 主构造器参数若不加 `val`/`var`，则默认为私有字段，子类无法直接访问
- 子类继承抽象类，**必须实现所有抽象成员**（除非子类也是抽象的）

> 💡 源码示例：`code4_19.scala` 演示了抽象类的定义与继承。

### Q5：什么是多态？多态存在的条件

**答：多态**指同一个类型的引用（父类或特质）在运行时可以指向不同子类的对象，调用相同方法名时表现出不同的行为（动态绑定）。

**必要条件：**
1. 存在**继承关系**（子类继承父类，或实现特质）
2. 子类**重写（`override`）** 父类的方法
3. **父类引用指向子类对象**（如 `val p: Point = new Dimension(...)`）

---

### Q6：特质（Trait）的作用

特质是 Scala 中**代码复用的核心机制**，主要作用：

- **定义接口** — 声明抽象方法，供实现类实现
- **提供默认实现** — 直接给出方法的默认实现
- **混入（Mixin）多个特性** — 一个类可同时混入多个特质，实现多重继承
- **实现依赖注入、装饰器模式**等高级设计（通过堆叠特质）

> 💡 源码示例：`code4_37.scala`、`code4_38.scala` 演示了 Trait 的混入使用。

---

### Q7：特质与类的相同点和不同点

| 对比维度 | 相同点 | 不同点 |
|:--------|:------|:-------|
| **成员定义** | 都可以有字段、方法、抽象成员 | 特质**不能有带参数构造器**；类可以有主/辅助构造器 |
| **继承** | 都可以被继承 | 类只能**单继承**（一个父类）；特质可以**多混入**（多个 `with`） |
| **实例化** | 都不能直接 `new` | 特质可通过匿名类实例化：`new Trait { ... }` |
| **初始化顺序** | — | 特质按**从左到右线性化初始化**，类父类先于子类 |

### Q8：Scala 中 `import` 关键字的功能

- **导入包或类** — `import scala.collection.mutable.ListBuffer`
- **导入对象所有成员** — `import math._`
- **重命名或隐藏**：
  - 重命名：`import java.util.{HashMap => JavaHashMap}`
  - 隐藏：`import java.util.{ArrayList => _, _}`

## 第五章 RDD编程

### Q1：什么是 Transformation 操作的惰性机制？

**答：惰性机制（Lazy Evaluation）** 指 Transformation 操作（如 `map`、`filter`）在被调用时**不会立即执行**计算，只构建一个**计算图（Lineage Graph）** 记录"血缘关系"。只有当 **Action 操作**（如 `collect`、`count`、`reduce`）被调用时，Spark 才真正触发计算。这种设计允许 Spark 对整个计算流程进行**全局优化**（如合并操作、避免不必要计算）。

> 💡 源码印证：`code1_1.scala:13-27` 中 `flatMap`、`map`、`reduceByKey` 都只是构建 DAG，直到 `collect()` 才真正运行。

---

### Q2：在依赖关系中，为什么说窄依赖要比宽依赖好？

**核心原因：窄依赖无需 Shuffle，宽依赖需要 Shuffle（跨网络传输数据）。**

| 对比维度 | 窄依赖 | 宽依赖 |
|:--------|:------|:------|
| **数据传输** | 同一节点内完成 | 需 **Shuffle**（跨网络） |
| **执行效率** | 支持**流水线执行**，效率高 | 需等所有父分区就绪，开销大 |
| **容错恢复** | 丢失分区可**快速**从父分区重建 | 恢复成本高，可能需重算多个分区 |
| **典型操作** | `map`、`filter`、`flatMap`、`union` | `reduceByKey`、`groupByKey`、`join` |

**一句话：窄依赖可以避免昂贵的 Shuffle 操作，显著提升性能。**

## 第六章 Spark SQL与DataFrame

### Q1：Spark SQL 有哪些特点？

- **易整合（Integrated）**：无缝整合 SQL 查询和 Spark 编程
- **统一数据访问（Uniform Data Access）**：相同接口连接 Hive、Parquet、JSON、JDBC 等
- **兼容 Hive（Hive Integration）**：直接运行 HiveQL
- **标准连接（Standard Connectivity）**：支持 JDBC/ODBC
- **高性能**：Catalyst 优化器自动优化（列裁剪、谓词下推），性能优于 RDD

> 💡 源码印证：`code6_1.scala` 展示了 SparkSession 创建和 `import spark.implicits._`。

### Q2：Spark SQL 支持哪些数据源？

| 类型 | 格式 | 短名称 |
|:----|:----|:------|
| **列式存储** | Parquet（默认）、ORC | `parquet`、`orc` |
| **文本格式** | JSON、CSV、TXT | `json`、`csv`、`text` |
| **关系数据库** | MySQL、PostgreSQL 等 | `jdbc` |
| **其他** | Hive、Avro、libsvm | `hive`、`avro`、`libsvm` |

### Q3：DataFrame 与 RDD 的区别是什么？

| 对比维度 | RDD | DataFrame |
|:--------|:---:|:---------:|
| **数据结构** | 无结构信息 | 带有 Schema（列名+类型） |
| **类型安全** | ✅ 类型安全 | ❌ 非类型安全 |
| **性能优化** | 简单流水线优化 | **Catalyst 优化器**（过滤下推、裁剪） |
| **适用场景** | 非结构化数据 | 结构化/半结构化数据 |
| **API 风格** | 函数式编程 | 类 SQL 声明式 API |

> DataFrame 在 Scala/Java 中实际是 `Dataset[Row]` 的类型别名。

## 第七章 Spark Streaming流式计算

### Q1：静态数据与流数据的区别及处理方法

| 数据类型 | 特点 | 处理方法 |
|:--------|:----|:--------|
| **静态数据** | 静止不动、相对稳定 | **批量计算**，充裕时间处理 |
| **流数据** | 大量、快速、持续到达，价值随时间流逝 | **实时计算**，到达立即处理 |

---

### Q2：什么是 DStream？

**答：** DStream（Discretized Stream，离散流）是 Spark Streaming 对流式数据的基本抽象。它将连续的数据流按**时间间隔**切分成一系列不连续的 RDD，本质上是一系列连续的 RDD 组成的序列。

```
DStream = RDD + 时间维度
```

---

### Q3：DStream 中按批处理间隔划分的元素与 RDD 的关系

**答：DStream 由一系列连续的 RDD 组成。** 每个**批处理间隔**内接收的数据对应生成一个 **RDD**。DStream 就是在 RDD 基础上增加了**时间维度**。

> 💡 源码印证：`code7_6.scala:7-9` 中 `new StreamingContext(conf, Seconds(10))` 设置批处理间隔为 10 秒。

---

### Q4：Spark Streaming 的三种基本输入源

| 输入源 | 写法 |
|:------|:-----|
| **文件系统** | `ssc.textFileStream(dir)` |
| **套接字连接** | `ssc.socketTextStream(host, port)` |
| **Akka Actor** | `ssc.actorStream(...)` |

> 扩展：Kafka 输入源（`KafkaUtils.createDirectStream`）也常用但不是"基本"输入源。

---

### Q5：DStream 的窗口操作是什么？

**答：** 窗口操作允许在数据流的**滑动窗口**上应用转换计算。通过设定**窗口长度**（看多久的数据）和**滑动间隔**（每隔多久算一次），让窗口按指定间隔在 DStream 上滑动计算。

```
时间轴: 0    10    20    30    40
窗口1: [============]           ← 第0~30秒
窗口2:       [============]     ← 第10~40秒
```

> 💡 源码印证：`code7_6.scala:9-13` 演示了 `words.window(Seconds(30), Seconds(10))`（窗口 30 秒，滑动 10 秒）。

---

### Q6：为什么要设置合适的批次大小？

批次大小（批处理间隔）是 Spark Streaming 的核心参数：

- 间隔**太短** → 数据未处理完，造成**延迟堆积**甚至 OOM
- 间隔**太长** → 无法满足**实时性**要求
- 建议从 5~10 秒开始测试，确保批处理时间不超过间隔

### Q7：有状态与无状态转换操作

| 类型 | 含义 | 典型操作 |
|:----|:----|:--------|
| **无状态转换** | 每个批次独立，不依赖之前数据 | `map`、`filter`、`reduceByKey` |
| **有状态转换** | 依赖之前批次的数据或中间结果 | **滑动窗口**、**`updateStateByKey`** |

## 第八章 Spark GraphX图计算

### Q1：GraphX 的基本数据结构包含哪些类型的 RDD？

GraphX 的**属性图（Property Graph）** 由两类 RDD 构成：

| 数据结构 | 说明 | 结构 |
|:--------|:----|:----|
| **VertexRDD[VD]** | 顶点表，存储顶点 id 及属性 | `(VertexId, VD)` |
| **EdgeRDD[ED]** | 边表，存储边及属性 | `(srcId, dstId, ED)` |
| **Triplet** | 逻辑视图，由前两者 join 得到 | `(srcAttr, dstAttr, edgeAttr)` |

> 💡 源码印证：`8_17.scala:14-30`（PageRank 示例：从 followers.txt 加载边数据）、`8_28.scala:13-31`（Twitter 图构建）。

---

### Q2：GraphX 的存储方式

GraphX 采用**分布式图存储 + 分割边结构**：

- 顶点（VertexRDD）分布式存储
- 边（EdgeRDD）按**分区策略（PartitionStrategy）** 存储
- 支持点分割（Vertex-cut），类似 PowerGraph 思想
- 顶点属性在各分区进行**缓存与索引**

**点分割 vs 边分割**：
| | 点分割（GraphX） | 边分割 |
|:--|:--------------:|:------:|
| 策略 | 每条边存一次，顶点可重复 | 每个顶点存一次，边可重复 |
| 原因 | 磁盘便宜，网络带宽贵，少传一次是一次 | 传统图计算框架 |

---

### Q3：GraphX 的实现架构

```
┌─────────────────────────┐
│   Graph API 层           │  ← Graph、Pregel API
├─────────────────────────┤
│   Graph Operator 层      │  ← mapVertices / mapEdges / aggregateMessages
├─────────────────────────┤
│   Execution Engine       │  ← 基于 Spark RDD DAG 调度
├─────────────────────────┤
│   Storage Layer          │  ← VertexRDD + EdgeRDD
└─────────────────────────┘
```

> **本质**：Spark RDD + 图计算封装层。

---

### Q4：aggregateMessages() 与 Pregel API 的异同

**相同点**：
- 都用于**图消息传递计算（Message Passing）**
- 都基于迭代计算模型
- 都可实现 PageRank / SSSP（单源最短路径）

**不同点**：

| 对比维度 | aggregateMessages | Pregel API |
|:--------|:----------------:|:----------:|
| 抽象层级 | 低层 | 高层 |
| 控制能力 | 更灵活 | 更封装 |
| 迭代机制 | 无内置迭代（需手动循环） | 内置 **superstep**（超步） |
| 使用复杂度 | 高 | 低 |

---

### Q5：PageRank 算法计算流程（GraphX）

1. **初始化**：每个节点 rank = 1.0
2. **迭代计算**：
   - 每个节点将 rank 均分给出边邻居
   - 接收邻居贡献：
   
     $$rank(v) = (1 - d) + d \times \sum \frac{rank(u)}{outDegree(u)}$$

3. **收敛条件**：迭代固定次数或误差 < ε
4. **GraphX 实现**：

```scala
graph.pageRank(0.0001).vertices
```

> 💡 源码印证：`8_17.scala:16` 完整演示了 PageRank 的执行。

---

### Q6：标签传播算法（Label Propagation）

1. 每个节点初始化一个 label（通常是自身 id）
2. 迭代过程：节点收集邻居 label，选择出现频率最高的 label
3. 收敛条件：label 不再变化

```scala
graph.labelPropagation.maxIter(10).run()
```

### Q7：Dijkstra 最短路径（带路径记录）

使用 VertexRDD 存储当前最短距离和路径列表：

1. 初始化 source = 0，其余 = ∞
2. 迭代松弛：`dist[v] = min(dist[v], dist[u] + w)`
3. 更新路径：`path[v] = path[u] + u → v`

> 在 GraphX 中通常用 **Pregel API** 实现。

### Q8：Kruskal 最小生成树算法实现思路

1. 将边按权重排序
2. 依次选择最小边
3. 使用并查集判断是否成环
4. 若不成环则加入 MST

> Spark 实现方式：EdgeRDD 按 weight `sortBy`，用 map + reduce 模拟 union-find。

```scala
  ("老年", "中", "是", "优", "是", 132),
  ("青年", "中", "否", "优", "是", 64),
  ("中年", "高", "否", "良", "是", 32),
  ("中年", "中", "否", "优", "是", 32),
  ("老年", "中", "否", "良", "是", 64)      // 补全最后一行
).toDF("age", "income", "student", "credit", "buy", "count")

// 2. 对所有特征列进行 StringIndexer
val featureCols = Array("age", "income", "student", "credit")
val indexers = featureCols.map { col =>
  new StringIndexer().setInputCol(col).setOutputCol(s"${col}_idx")
}
val labelIndexer = new StringIndexer()
  .setInputCol("buy").setOutputCol("label")

// 3. 组装特征向量
val assembler = new VectorAssembler()
  .setInputCols(Array("age_idx", "income_idx", "student_idx", "credit_idx"))
  .setOutputCol("features")

// 4. 决策树分类器（带权重）
val dt = new DecisionTreeClassifier()
  .setLabelCol("label").setFeaturesCol("features")
  .setWeightCol("count")    // ⚠️ 关键：使用计数作为样本权重
  .setImpurity("gini").setMaxDepth(5)

// 5. Pipeline 训练
val pipeline = new Pipeline().setStages(indexers ++ Array(labelIndexer, assembler, dt))
val model = pipeline.fit(data)

// 6. 输出决策树结构
val treeModel = model.stages.last
  .asInstanceOf[org.apache.spark.ml.classification.DecisionTreeClassificationModel]
println("决策树结构:\n" + treeModel.toDebugString)
```

## 附录：核心易错点速查

1. **元组从 1 开始**（`t._1`），数组/列表从 0 开始
2. **`===` 不是 `==`**（DataFrame 的 join 条件用 `===`）
3. **辅助构造函数第一行必须 `this()`**
4. **`reduceByKey` 是 T 不是 A**（所有 ByKey 结尾的都是 Transformation）
5. **DataFrame 要 `import spark.implicits._`**
6. **Streaming 的 `local[2]`** — 至少 2 线程（1 收 1 处理）
7. **窗口高效版和 `updateStateByKey` 都要 checkpoint**
8. **顶点 ID 是 Long 类型**（后面加 `L`，如 `1L`）
9. **`override`** — 重写非抽象方法必须加
10. **if-else 在 Scala 里是表达式，有返回值**
