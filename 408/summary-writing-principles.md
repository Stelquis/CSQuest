# 408 课程总结文档编写原则

> 版本：v1.0
> 用途：为数据结构、计算机组成原理、操作系统、计算机网络四门课，编写"故事型"课程总结的方法论
> 核心理念：**不讲教材目录，讲一个"被逼出来"的故事**

---

## 原则一：一条线到底，不设章节号

### 不要

```
第一章：XXX
第二章：XXX
```

或者

```
第一幕：XXX
序幕：XXX
```

### 要

从头到尾一个故事，用**自然过渡**衔接。每个段落结尾自然引出下一个段落。

**过渡句式**（不用"本章总结了……"之类的教材语言）：

| 过渡方向 | 自然句式 |
|:---------|:---------|
| 引出问题 | "但问题来了：……" |
| 引出方案 | "所以需要……" |
| 引出代价 | "但代价是……" |
| 引出进化 | "但还不够——……" |
| 引出瓶颈 | "但瓶颈在于……" |

**文章开头**：从一个**具体的、原始的需求**出发，不要从"计算机系统概述"这种抽象概念开始。

> ✅ 好开头："有一个人，他想算东西。"
> ❌ 差开头："计算机系统由五大部件组成……"

### 原则一的应用示例

```
（同一段末尾）……至此，CPU 能算也能记了。但问题来了：CPU 算一次几纳秒，
去主存取数据却要几十纳秒——CPU 大部分时间在"等"。

（下一段开头）所以需要一种"快到能跟上 CPU、又大到能存够数据"的存储器。
于是 Cache 诞生了……
```

---

## 原则二：每个概念都是"补丁"，不是"发明"

### 核心思想

**不要**把技术当作一个"天才的发明"来介绍，要当作"被逼出来的补丁"来描述。

**每个概念回答三个问题**：
1. 之前有什么？—— 原来的状态
2. 它哪里不够？—— 具体的矛盾
3. 所以造了什么？—— 解决方案

### 不要

```
Cache 是一种高速缓存存储器，位于 CPU 和主存之间……
（平铺直叙，像抄词典）
```

### 要

```
CPU 算一次只要几纳秒，但去主存取数据要几十纳秒。
CPU 大部分时间在"等主存"——这就是"存储墙"问题。
太慢了。所以需要在中间加一层快的：Cache。
```

### 检查清单

写完后，对每个核心概念问自己：

- [ ] 读者知道"为什么要有这个东西"吗？
- [ ] 读者知道"没有它之前，哪里不够用"吗？
- [ ] 读者知道"它解决了问题，但带来了什么新问题"吗？

---

## 原则三：不贴标签，不用"故事"这个字眼

### 不要

```
故事：……
第一幕：……
```

这些标签**提醒读者"这是一个故事"**，反而破坏了沉浸感。

### 要

直接讲。让读者**自己感觉到**这是一个故事，而不是你告诉他"这是一个故事"。

**用词偏好**：

| 不要用 | 用 |
|:-------|:---|
| "本章介绍了……" | 直接讲技术本身 |
| "故事是这样的……" | 直接叙述 |
| "第一幕/第二幕" | 空行分隔 + 自然过渡 |
| "让我们来总结一下" | 自然收束 + 引出下一个问题 |

---

## 原则四：从"原始需求"出发，不是从"定义"出发

### 每门课的开头

| 科目 | 原始需求（起点） |
|:-----|:----------------|
| 数据结构 | 有数据要存，有操作要做——怎么组织最高效？ |
| 计算机组成原理 | 有一个人，他想算东西。 |
| 操作系统 | 有了硬件，但人不能直接操作硬件——需要一个"管家"。 |
| 计算机网络 | 两台计算机之间要传数据。 |

**每一门课的第一个问题，必须是"人为什么要做这件事"**，而不是"这个学科的定义是什么"。

---

## 原则五：每个阶段结束要"全景回看"

虽然不使用章节号，但在几个关键节点后，需要停下来让读者看到"现在计算机长什么样了"。

### 全景回看的方式

不是"总结本章要点"，而是**指着当前已经造出来的部件说"现在它长这样"**。

例如：

```
至此，我们有了：
  ├── 能算数的 ALU
  ├── 能存数的寄存器 + 主存
  ├── 能加速的 Cache
  ├── 能听懂指令的指令系统
  └── 能执行指令的 CPU
问题：这些部件之间怎么通信？
```

**全景回看的位置**：大约每讲完 2-3 个"补丁"后回看一次，不要每段都回看。

---

## 原则六：保留"致命陷阱"和"跨科缝合"，但融入叙事

### 致命陷阱

不要单独列一个"致命陷阱清单"放在最后，而是**在讲到对应概念时自然插入**。

```
……补码的溢出判断要看最高位和次高位的进位。这是选择题最爱挖坑的地方——
两个同号数相加，结果符号变了，就是溢出。
```

### 跨科缝合

在合适的叙事节点，自然引出"这个技术在其他科目也有对应"。例如在讲到 Cache 替换算法时：

```
这种"把最常用的留在身边"的思路，在操作系统的页面置换里也会出现——
LRU 算法在计组和 OS 里都是"淘汰最久没用的那个"。
```

---

## 原则七：叙事节奏——"三段式"推进

每个"补丁"的叙事节奏：

```
┌─────────────────────────────────────┐
│ 第一步：现状                         │
│ "现在我们已经有了 A。"                │
│                                     │
│ 第二步：A 的瓶颈                     │
│ "但 A 不够用，因为……"                 │
│                                     │
│ 第三步：解决方案 B                   │
│ "所以需要 B，它的原理是……"            │
│                                     │
│ 第四步：B 的代价（可选，重要概念保留） │
│ "但 B 也有代价——……"                  │
│                                     │
│ （自然过渡到下一个"补丁"）            │
└─────────────────────────────────────┘
```

## 原则八：语言风格

### 口语化，有画面感

- 多用**类比**：用生活场景解释技术概念
- 多用**短句**
- 少用**被动语态**
- 少用**抽象概念开头**

### 避免

```
计算机组成原理研究的是计算机内部各部件的组织结构和工作原理……
（像一个考试大纲）
```

### 追求

```
你想算一个数。交给 CPU，CPU 说"好，但我要先去内存拿数据"。
去内存拿数据要多久？几十纳秒。
CPU 算一下要多久？几纳秒。
CPU 大部分时间在等内存。
```

### 类比库（日常积累）

| 技术概念 | 生活类比 |
|:---------|:---------|
| 补码（模运算） | 时钟：减 2 小时 = 加 10 小时（模 12） |
| Cache | 桌面 vs 书架：最常用的书放桌上 |
| 中断 | 健身房接电话：停下来接，接完继续 |
| DMA | 快递员搬货：你继续做自己的事 |
| 总线 | 一条马路，多个路口 |
| 流水线 | 工厂流水线：每个工位做一道工序 |
| 虚拟内存 | 酒店房间钥匙：每个客人以为有自己的房间 |

---

## 原则九：篇幅控制

| 内容 | 建议篇幅 | 说明 |
|:-----|:--------:|:-----|
| 完整总结文档 | 5000-12000 字 | 对应一门课，覆盖 80% 核心概念 |
| 每个"补丁" | 300-800 字 | 一个技术点从"问题"到"解决" |
| 全景回看 | 100-200 字 | 快速勾勒当前全貌 |

---

## 原则十：最终输出格式

### 标题

```
# 科目名：一句话概括整个故事
```

例如：

```
# 计算机组成原理：从"算数"到"计算机"的进化史
# 数据结构：数据怎么组织，操作才最快？
# 操作系统：一个"管家"的自我修养
# 计算机网络：让两台计算机"说话"
```

### 段落

- 段落之间用空行分隔
- 关键概念可以用**加粗**强调
- 复杂的技术可以用表格辅助说明，但**不要用表格代替叙事**
- 代码块或公式块可以嵌入，但不要打断叙事节奏

### 禁用清单

- ❌ 章节编号（第一章、第 2 章）
- ❌ "本章介绍了"、"本节讲述了"、"综上所述"
- ❌ "故事："、"序幕"、"第一幕" 等标签
- ❌ "让我们来总结一下"、"通过本文的学习"
- ❌ 教材式定义开头（"XXX 是一种……"）

---

## 附录：快速参考卡

### 写作前问自己

```
1. 这门课最初要解决什么问题？ → 开头
2. 每个概念是"不够"了什么才出现的？ → 叙事线
3. 读者看完能不能"重走一遍发明过程"？ → 检查标准
```

### 写完后的检查清单

- [ ] 全文是一条线，没有跳脱感？
- [ ] 每个概念出现前，读者已经知道"为什么需要它"？
- [ ] 没有使用教材套话（"本章"、"综上所述"）？
- [ ] 没有使用"故事"标签？
- [ ] "致命陷阱"融入了叙事，没有单独列在最后？
- [ ] 跨科缝合点自然插入，而不是单独列在最后？
- [ ] 开头是从"原始需求"出发，而不是从"定义"出发？
- [ ] 读起来像"在看一个人从零造东西"，而不是"在看教材目录"？
