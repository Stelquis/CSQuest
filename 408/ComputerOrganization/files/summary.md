# 计算机组成原理：从"算数"到"计算机"的进化史

> 这不是一本教材的目录，这是一台计算机从零开始"被迫进化"的全过程。
> 每一段都回答一个问题——上一个问题的答案，恰好是下一个问题的起点。
>
> 来源：9 篇微信公众号文章（罗念笙），原文约 8 万字
> 整理日期：2026-07-17

---

有一个人，他想算东西。

一开始他用算盘，但算盘要手拨，太慢。后来有了机械计算器，但它只能做一种运算——换个算法就得换机器。太麻烦了。

他想要一个东西：**插上电，告诉它"算什么"，它就自动算完**。而且今天算加法，明天算乘法，不用换机器。

这个想法，就是计算机的起点。

---

## 一、从"人的数学"到"机器的数学"

最直接的问题：人用十进制（0-9），但电路只有两种状态——通和断。

这意味着一切的起点是：**怎么用 0 和 1 来表示数字？**

正数好办——直接转二进制就行。但负数呢？小数呢？

### 原码——最直觉的方案

最高位表示符号（0 正 1 负），其余位表示绝对值。人一眼能看懂，但做加法的时候麻烦就来了：先判断两个数的符号，再决定是加还是减，最后还要决定结果的正负。**一个加法器搞不定，加减法电路要分开设计。**

### 补码——减法变加法

补码的灵感来自时钟：3 点调到 1 点，可以倒拨 2 小时（3-2=1），也可以正拨 10 小时（3+10=13≡1 mod 12）。**在模 12 下，减 2 等价于加 10。**

计算机也一样。n 位二进制就是模 2ⁿ 的运算。一个负数的补码，就是它正数的"补数"——取反加一。

**好处是什么？**加减法统一了。**一个加法器就够了，不需要专门的减法器。**符号位也参与运算，溢出看最高位和次高位的进位是否一致。

### 移码——浮点数的铺垫

移码 = 补码的符号位取反。它存在的唯一原因：**方便浮点数比较大小**——全 0 最小，全 1 最大，可以直接按无符号数比较。

### 浮点数——科学计数法的二进制版

整数够了，但现实世界的数据大多是小数——温度、距离、价格。

最直接的想法是**定点数**：约定前 16 位是整数，后 16 位是小数。但问题来了——整数部分多了小数没精度，小数部分多了整数范围不够。

**IEEE 754 浮点数**的答案：像科学计数法一样，用指数来"浮动"小数点。

```
单精度 32 位：1 位符号 + 8 位阶码 + 23 位尾数
公式：(-1)^S × 1.M × 2^(E-127)
```

规格化数要求尾数以 1. 开头（隐含不存），最大化了精度。

但浮点数有一个**致命陷阱**：**只有 2 的幂次组合能精确表示**（0.5、0.25、0.125……）。0.1 在二进制中是无限循环小数，所以 0.1 + 0.2 ≠ 0.3。这不是 bug，是二进制表示的固有缺陷。

### ALU——谁来算？

有了数字表示，还需要一个"算数"的电路——**ALU（算术逻辑单元）**。

它是组合逻辑电路，输入两个 N 位二进制数，输出运算结果。能做加减乘除、与或非、移位。

**加法器的进化**：从串行加法器（一位一位加）到并行加法器（一起加但进位逐位传递）到先行进位加法器 CLA（提前算好进位，速度快但电路复杂）。

**两种移位**：算术移位（符号位不变，空位补符号位，相当于乘/除 2）和逻辑移位（空位补 0，相当于无符号数乘/除 2）。

---

**到此，我们有了一个能"算"的机器。**

但问题来了：ALU 算完一个数，结果如果不存起来，下一个运算就把它覆盖了。**机器算完就忘**——这不行。

---

## 二、机器需要"记忆"

### 寄存器——最快但最贵的记忆

最直接的想法：在 CPU 内放一些**寄存器**，用来存中间结果。速度快到能跟上 ALU，但代价是——**贵**。一个寄存器几十个晶体管，CPU 芯片就那么点面积，放不了太多。

放不下大量数据怎么办？那就单独做一个**专门用来存数据**的部件。

### 主存——大但慢

**动态 RAM（DRAM）** 用电容来存储电荷，有电代表 1，没电代表 0。一个电容加一个晶体管就能存一位，集成度高，价格低。

但代价是：
- **慢**：DRAM 的访问时间几十纳秒，而 CPU 执行一条指令只要几纳秒
- **需要刷新**：电容会漏电，每隔几毫秒就要重新充一次电（刷新）

**三种刷新方式**：
- **集中式**：集中一段时间刷新所有行——死区时间 64μs，期间不能访存
- **分散式**：每读/写一次就刷新一行——没有死区，但存取周期加倍
- **异步式**：在译码阶段刷一行——折中方案，最常用

**SRAM 和 DRAM 的分工**：
- **SRAM（静态 RAM）**：用触发器存储，速度快，但集成度低、贵 → 做 Cache
- **DRAM（动态 RAM）**：用电容存储，速度慢，但集成度高、便宜 → 做主存

### 存储器扩展——芯片不够用

一片存储芯片的字长和容量是固定的，但实际需求多种多样。

- **位扩展**：增加字长（8 片 1M×1 位的芯片拼成 1 片 1M×8 位）
- **字扩展**：增加容量（4 片 1M×8 位的芯片拼成 4M×8 位）
- **字位扩展**：同时增加字长和容量

### 海明码——数据会出错

DRAM 的电容会漏电，alpha 粒子可能翻转存储单元。数据在存储和传输过程中可能出错。

**海明码**的解决思路：在数据位中插入校验位，每个校验位负责一组数据位。出错时，多个校验位会"举报"错误的位置。

公式：2^r ≥ m + r + 1（m 是数据位数，r 是校验位数）。8 位数据需要 4 位校验，能纠 1 位错或检 2 位错。

---

**现在，我们有了能算的 ALU，能存的主存。但问题更大了：**

CPU 算一次几纳秒，去主存取数据却要几十纳秒。**CPU 大部分时间在"等"——等主存把数据送过来。**

这就是"存储墙"问题。

---

## 三、主存太慢——Cache 来了

### 局部性原理——Cache 能工作的唯一理由

如果程序每次都随机访问不同地址，Cache 毫无意义——命中率是 0%。

但程序不是随机的，它有两个规律：
- **时间局部性**：刚访问过的数据，很可能很快再次访问（循环体里的变量）
- **空间局部性**：刚访问过的数据附近的数据，很可能很快被访问（数组的顺序遍历）

### 怎么知道数据在不在 Cache 里？

CPU 要访问一个内存地址，它怎么知道这个地址的数据有没有被 Cache 存起来？

**三种映射方式，从"简单粗暴"到"灵活但费钱"**：

**直接映射**：每个内存块只能去 Cache 的一个固定位置。查起来最快——算一下就知道在不在。但冲突率最高——多个内存块抢同一个位置，频繁替换。

**全相联映射**：每个内存块可以去 Cache 的任何位置。最灵活，但查起来最慢——要遍历所有 Cache 行才能知道数据在不在。

**组相联映射**：折中——先分成组，每个内存块只能去固定组，但组内可以随意放。

**地址拆分**（组相联的）：**标记位（高位）| 组号（中间）| 块内地址（低位）**——这是选择题最爱挖坑的地方。

### Cache 满了怎么办？

新数据要进来，旧数据谁滚蛋？

- **FIFO**：先进先出——简单但可能把刚用到的数据踢出去
- **LRU（近期最少使用）**：淘汰最久没被访问的——命中率最高，但硬件实现复杂（需要记录访问顺序的顺序栈）
- **随机**：随便选一个——命中率最低，但硬件最简单

### Cache 里的数据和主存不一致了怎么办？

Cache 改了，主存要不要同步改？

- **写直达**：写 Cache 的同时也写主存——数据一致性好，但慢（每次写都要等主存）
- **写回法**：只写 Cache，标记"脏位"，等替换的时候再写回主存——快，但数据可能不一致

### 性能公式

平均访问时间 = 命中率 × Cache 时间 + 未命中率 ×（主存时间 + Cache 时间）

$T_{avg} = h \cdot t_c + (1-h) \cdot (t_m + t_c)$

**现实例子**：h=95%，t_c=1ns，t_m=50ns → T_avg=3.5ns。如果没有 Cache，直接访存=50ns。**快了 14 倍。**

---

**有了 Cache，CPU 不用再干等主存了。但新的问题出现了：**

我们有了能算的 ALU、能记的寄存器、能大量存的主存、能加速的 Cache。但怎么告诉这台机器"你要做什么"？

---

## 四、怎么告诉机器"做什么"

人说话是"加 3 和 5"，但机器只懂 0 和 1。需要一个**翻译规则**——这就是**指令系统**。

### 一条指令长什么样？

```
操作码（做什么）+ 地址码（对谁做）
```

操作码告诉 CPU 要做什么（加、减、读、写、跳转），地址码告诉 CPU 操作数在哪。

### 操作数藏在哪？——寻址方式

操作数可能藏在各种地方——常数、变量、指针、数组、循环……每种情况需要不同的**寻址方式**：

- **立即寻址**：操作数就在指令里——"就是它"（常数）
- **直接寻址**：操作数在内存的某个地址——"就在这个地址"（全局变量）
- **间接寻址**：地址里存着地址——"一层一层找"（指针）
- **寄存器寻址**：操作数在寄存器里——"最快"（中间变量）
- **寄存器间接寻址**：寄存器里存着地址——"通过寄存器找内存"（指针变量）
- **基址寻址**：基地址 + 偏移——"程序重定位"（OS 管理基址寄存器）
- **变址寻址**：基地址 + 索引——"数组遍历"（用户管理变址寄存器）
- **相对寻址**：当前 PC + 偏移——"跳转指令"
- **堆栈寻址**：隐含使用 SP——"函数调用"

**基址 vs 变址**：基址是 OS 管的（程序重定位），变址是用户管的（数组遍历）。考题常在这里挖坑。

### 指令越多越好还是越少越好？

这是计算机历史上的一场大争论。

**CISC（复杂指令集）** 的哲学：指令越强大越好，一条指令能完成复杂操作。x86 是代表。优点是汇编编程简单，代码密度高。缺点是译码复杂，不利于流水线。

**RISC（精简指令集）** 的哲学：指令越简单越好，复杂的操作由多条简单指令组合。ARM、MIPS、RISC-V 是代表。优点是固定指令长度，译码简单，容易实现流水线。缺点是代码密度低。

| 特性 | CISC | RISC |
|:----|:----|:----|
| 指令条数 | 多（几百条） | 少（几十条） |
| 指令长度 | 可变 | 固定 |
| 通用寄存器 | 较少 | 多（32+） |
| 控制方式 | 微程序控制 | 硬布线控制 |
| 访存指令 | 大多可直接访存 | 只有 load/store |
| 典型架构 | x86 | ARM、MIPS、RISC-V |

---

**有了指令系统，我们终于能告诉机器"做什么"了。但谁来"读"指令、"翻译"指令、"执行"指令？**

需要一个大脑。

---

## 五、大脑——CPU

### CPU 的两大核心

**运算器（数据通路）**：负责算。核心部件是 ALU、ACC（累加器）、MQ（乘商寄存器）、X（操作数寄存器）。

**控制器（控制通路）**：负责指挥。核心部件是：
- **PC（程序计数器）**：指向下一条要执行的指令的地址
- **IR（指令寄存器）**：存放当前正在执行的指令
- **CU（控制单元）**：解读 IR 中的指令，产生控制信号

### 一条指令的一生——指令周期

**取指**：PC → MAR → 主存 → MDR → IR，PC 自增
**间址**（可选）：如果需要间接寻址，再访存一次得到有效地址
**执行**：ALU 运算 / 访存 / 跳转
**中断**（可选）：检查是否有中断请求，保存断点

### 顺序执行太慢了——流水线

**假设**：每条指令需要 5 个阶段，每个阶段 1ns。
- 非流水线：执行 N 条指令要 5N ns，每 5ns 完成一条
- 流水线：执行 N 条指令要 5 + (N-1) ns，每 1ns 完成一条

**理想加速比 = 流水线段数 = 5 倍。**

但理想很丰满，现实很骨感。

### 流水线冲突——为什么不能一直 1ns 一条

**结构冲突**：多条指令同时抢同一个硬件资源。比如 IF 要取指令，MEM 要访存——如果指令 Cache 和数据 Cache 是同一个，就打架了。**解决**：分离指令 Cache 和数据 Cache，或者插入气泡。

**数据冲突**：下一条指令依赖上一条的结果，但结果还没写回来。比如：
```
ADD R1, R2, R3    ; R1 = R2 + R3
SUB R4, R1, R5    ; R4 = R1 - R5（需要 R1，但 R1 还没算完）
```
**解决**：转发（旁路）——把 ALU 的结果直接送到下一条指令的 ALU 输入，不经过寄存器。但如果转发来不及（比如 load 指令后面紧跟使用该数据的指令），必须插入气泡。

**三种数据冲突**：RAW（写后读，真依赖，最常见）、WAR（读后写，反依赖）、WAW（写后写，输出依赖）。后两种在流水线中很少出现（因为有大量寄存器）。

**控制冲突**：分支指令（if-else、循环）改变了执行顺序，但流水线已经预取了后面的指令。猜错了，已经进流水线的指令全部作废。**解决**：分支预测（猜对了继续，猜错了清空）、延迟分支（在分支后面放一条"无论跳不跳都执行"的指令）。

### 流水线的进阶

- **超标量**：一个时钟周期发射多条指令——多条流水线并行
- **超流水线**：把流水线分成更多更细的阶段——提高主频
- **VLIW**：编译器打包多条指令，硬件一次性执行——硬件简单，编译器复杂

### 控制单元——怎么"指挥"整个 CPU

**问题**：有了指令，有了数据通路，谁来产生"什么时候、哪个部件、做什么"的控制信号？

**方案一：硬布线控制**——用逻辑门电路直接产生控制信号。**快，但死板**。适合 RISC（指令少，逻辑简单）。

**方案二：微程序控制**——把控制信号事先存在一个 ROM 里，执行指令时读出来。**灵活，但慢**。适合 CISC（指令复杂，控制逻辑多）。

| 特性 | 硬布线 | 微程序 |
|:----|:------:|:------:|
| 速度 | 快 | 慢 |
| 灵活性 | 差 | 好 |
| 修改难度 | 难（改硬件） | 易（改控制存储器） |
| 适用 | RISC | CISC |

**微指令的两种格式**：水平型（每个控制信号占一位，并行能力强，字长长）和垂直型（用编码表示控制信号，并行能力弱，字长短）。

---

**到此，我们有了一个完整的"大脑"——能读指令、能执行、能流水线加速。**

但问题来了：CPU 要和内存通信，要和 I/O 设备通信，它们之间怎么连接？

---

## 六、怎么让所有部件"说话"

最简单的方案：每个设备之间拉一根专用的线。

但 n 个设备需要 n(n-1)/2 条线——**指数级增长，不可行。**

于是有了**总线**——一条共享通道，所有设备都连上去，分时使用。

### 总线的分类

- **片内总线**：CPU 芯片内部
- **系统总线**：计算机内部，连接 CPU、内存、I/O
- **通信总线**：计算机之间（USB、PCIe）

**系统总线**又分三类：**数据总线**（双向，宽度决定一次传输多少数据）、**地址总线**（单向，CPU→其他，宽度决定最大寻址空间——32 位→4GB，64 位→16EB）、**控制总线**（双向，传输控制信号）。

### 总线结构——从单车道到多车道

**单总线**：所有设备挂同一条总线。结构简单，但**同一时刻只能有一对设备通信**，总线是瓶颈。

**多级总线**：高速设备靠近 CPU，低速设备远离 CPU。CPU ↔ Cache ↔ 系统总线 ↔ 主存 ↔ 扩展总线 ↔ I/O 设备。

### 总线仲裁——谁先说话

多个设备同时想占用总线，谁先谁后？

**链式查询**：总线授权信号从高优先级设备依次传递。**简单，但低优先级设备可能"饿死"，链断了就全瘫。**

**计数器定时查询**：CPU 用计数器遍历设备编号，被"点到"的如果请求总线就获得使用权。**优先级可编程，设备不会饿死。**

**独立请求**：每个设备有独立的请求线和授权线，CPU 内部仲裁。**最快，但需要大量线。**

| 方式 | 速度 | 公平性 | 硬件复杂度 |
|:----|:----:|:------:|:----------:|
| 链式查询 | 慢 | 差 | 低 |
| 计数器定时查询 | 中 | 中 | 中 |
| 独立请求 | 快 | 好 | 高 |

### 总线通信——怎么说话

**同步通信**：所有设备按统一时钟通信。**简单，但所有设备必须按最快的速度工作。**

**异步通信**：没有统一时钟，通过握手信号协调。**灵活，但协议复杂。** 三种握手方式：不互锁（最快但不可靠）、半互锁、全互锁（最可靠但最慢）。

**半同步通信**：同步 + 异步结合——基本按同步，但允许慢设备插入"等待"信号。

**分离式通信**：把一次传输分成两个阶段——主设备发请求后释放总线，从设备准备好后重新申请总线发送数据。**关键思想：从设备准备数据期间，总线可以被其他设备用。**

---

**至此，CPU、内存、I/O 设备之间能通信了。**

但还有一个更大的问题没解决：计算机不能活在真空中，它需要和人交互——键盘、鼠标、显示器、磁盘……

但 CPU 是纳秒级的，外设是毫秒级的——**速度差了几百万倍。**

---

## 七、高速 CPU 怎么等得起慢速外设

### 方法一：程序查询——最笨的办法

CPU 不断查外设的状态寄存器——"好了没？好了没？好了没？"

**类比**：你在餐厅门口一直问"好了没？"——服务员没空干活，全在回答你。

CPU 利用率极低，大部分时间在空转。适合低速设备，但仅此而已。

### 方法二：中断——设备主动通知

CPU 正常执行程序，外设准备好后**主动发中断信号**，CPU 暂停当前程序，执行中断服务程序，完成后恢复。

**流程**：外设发请求 → CPU 当前指令完成后响应 → 保存断点 → 取中断向量 → 执行中断服务程序 → 恢复现场 → 返回

**类比**：你在健身房锻炼，手机响了（中断），你停下来接，接完继续练。

**多个设备同时发中断，谁先被处理？**
- 硬件排队：按优先级连接中断请求线
- 中断屏蔽字：每个中断源对应一位，1 表示屏蔽。**高优先级可以屏蔽低优先级。**

**中断响应优先级 vs 中断处理优先级**：响应优先级是硬件决定的（谁先被响应），处理优先级是通过屏蔽字设置的（谁能打断谁）。

但中断也有代价：**每次中断都要保存/恢复现场**。如果中断频率太高，CPU 花在中断上的时间反而比花在主程序上的还多。

### 方法三：DMA——CPU 不参与

**DMA 控制器**接管总线，直接在外设和内存之间传输数据，CPU 完全不参与。

**流程**：CPU 设置 DMA 控制器（传输方向、起始地址、字节数）→ DMA 开始传输，CPU 继续做自己的事 → 传输完成，DMA 发中断通知 CPU。

**类比**：你让快递员（DMA 控制器）去搬货，你继续做自己的事。搬完了快递员告诉你一声。

**DMA 访问主存的三种方式**：
- 周期挪用：DMA 在 CPU 不访存时"偷"一个总线周期——对 CPU 影响最小
- 停止 CPU 访问：DMA 传输期间禁止 CPU 访存——快，但 CPU 完全暂停
- 交替访问：CPU 和 DMA 轮流访存——需要双端口存储器

| 方式 | CPU 参与 | 速度 | 适合 | 本质 |
|:----|:--------:|:----:|:-----|:-----|
| 程序查询 | CPU 全程守护 | 慢 | 低速设备 | "主动问" |
| 程序中断 | CPU 响应后介入 | 中 | 中速设备 | "等通知" |
| DMA | 只是设置一下 | 快 | 高速设备 | "放权" |

### I/O 接口——外设和主机之间的"翻译官"

为什么需要接口？因为外设和主机之间：
- 速度不同 → 需要缓冲
- 信号电平不同 → 需要转换
- 数据格式不同 → 需要转换
- 工作方式不同 → 需要控制

一个 I/O 接口包含：数据线、设备选择线、命令线、状态线。

## 全景回看

至此，一台完整的计算机诞生了。

让我们停下脚步，看看我们这一路造了什么：

```
人想算东西
    ↓
① 数字表示：原码 → 补码 → 浮点数（解决"怎么用 0 和 1 表示数字"）
    ↓
② ALU（解决"谁来算"）
    ↓
③ 寄存器 + 主存（解决"算完放哪"）
    ↓
④ Cache（解决"主存太慢"）
    ↓
⑤ 指令系统（解决"怎么告诉机器做什么"）
    ↓
⑥ CPU + 流水线（解决"谁来执行指令"和"太慢"）
    ↓
⑦ 总线（解决"部件之间怎么通信"）
    ↓
⑧ 中断 + DMA（解决"CPU 和慢速外设怎么协作"）
```

**每一步都是因为"不够"而被迫进化。** 没有一步是"天才的发明"——都是"被逼出来的补丁"。

## 那些最容易被坑的地方

**1. 补码溢出判断**：两个同号数相加，结果符号变了 → 溢出。不是看最高位有没有进位。

**2. 浮点数精度**：0.1 + 0.2 ≠ 0.3。只有 2 的幂次组合能精确表示。这不是 bug，是二进制的固有缺陷。

**3. Cache 地址拆分**：组号在中间，标记在高位，块内地址在低位。不是"组号在前"——这是选择题最喜欢挖的坑。

**4. LRU 的实现**：用顺序栈，不是时间戳。每次访问一个 Cache 行，把它移到栈顶。替换时选栈底。

**5. 数据冲突 RAW**：后面指令读的寄存器，前面指令还没写完。这是流水线数据冲突中最常见的类型。

**6. 分支预测失败**：预测失败时，已经进入流水线的指令全部作废。**penalty = 清空流水线的代价。**

**7. 微程序 vs 硬布线**：微程序 = 软件思路（控制信号存在 ROM 里），硬布线 = 硬件逻辑门（直接连线）。**不是"微程序比硬布线先进"——它们只是适用于不同的场景。**

**8. 中断屏蔽字**：1 表示屏蔽，0 表示开放。**注意和直觉相反——"屏蔽"是禁止，不是允许。**

## 跨科缝合：计组和其他三科的关系

**Cache ↔ 页表（OS）**：Cache 用组相联映射把内存地址翻译成 Cache 位置，页表用多级映射把虚拟地址翻译成物理地址。**思路一模一样——都是"映射"解决"不够"的问题。**

**LRU 替换算法（Cache/OS）**：Cache 里 LRU 用硬件顺序栈实现，OS 里 LRU 用软件链表实现。**思路一样，实现不同。**

**指令流水线 ↔ 进程上下文切换（OS）**：流水线冲刷 = 进程切换的硬件代价。**每次切换进程，流水线里的指令全部作废。**

**中断 ↔ 系统调用（OS）**：中断是 I/O 设备通知 CPU 的方式，系统调用是程序请求 OS 服务的方式。**底层机制是一样的——都是"当前的程序停下来，去处理别的事"。**

**数据表示 ↔ 位图（数据结构）**：位图就是用二进制位表示集合——每个位表示一个元素是否存在。**和数据表示中的"用有限位表示信息"是同一个思想。**
